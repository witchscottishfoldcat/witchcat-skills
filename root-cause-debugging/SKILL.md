# 根因排障工作流

这是一套**通用排障方法论**，不是某个项目的流程。核心信念：**症状不是根因；
别凭症状或一个看起来合理的假设就改代码。每个结论都要先复现、再沿代码追到机制，
修完要在"暴露这个 bug 的同款条件"下验证，并扫一遍同类拷贝。**

贯穿全程的判断原则（比任何单步都重要）：

1. **复现先于诊断，诊断先于修改**——没复现就不算定位。能从真实环境复现的，
   绝不靠脑补；本地复现必须用**生产同款条件**（环境变量、请求头、Host、绑定地址、构建产物）。
2. **给机制级根因，不给"看起来像"**——能说清"哪一行代码、在什么条件下、为什么"才算找到根因。
   "可能是缓存问题"不是根因，"`headers()` 抛的 DynamicServerError 被 try/catch 吞掉导致页面被静态固化"才是。
3. **区分表象与机制，敢推翻表面判断**——"切换中文 URL 变了但页面还是英文"的表象是前端路由，
   机制却在构建期固化；"赛程没国旗"的表象是缺逻辑，机制却是图片 URL 404。用户的怀疑方向常对、
   细节常错——核实后明确指出哪里对、哪里不对、以及有没有更糟的情况。
4. **一个 bug 往往有多份拷贝**——同样的错误模式常散落在多个文件/多个调用点。修完必须 `grep`
   同一模式，连带相关的失效/缓存/同步/兜底逻辑一起改，否则只修了你撞见的那一处。
5. **最小爆炸半径**——优先改根因点而非到处加补丁；保留原有行为；加超时/兜底而非删功能。
6. **诚实交付边界**——说清"修了什么、为什么之前没发现、还有什么前提/限制"，不夸大、不隐瞒残留风险。

## 阶段 1 · 复现（从真实环境取证）

目标：把"用户描述的症状"变成"我亲眼看到的、可稳定触发的现象"。

操作：
- **直接打真实端点**，用 `curl -s -D -`（看状态码+响应头）对比**不同输入**下的差异——这是最快的定位器：
  - 不同 locale：`-H "Accept-Language: zh"` vs `en`，看是否同样坏（→ 与语言无关 = 排除 i18n）
  - 不同路径/参数：坏的接口 vs 好的接口（`/api/a` 500 但 `/api/b` 200 → 缩小到 a 的特有代码）
  - 不同伪造头：`-H "X-Forwarded-For: 9.9.9.9"`，看限流/鉴权/真实 IP 是否被影响
  - 重定向：`-D -` 看 `Location` 是否指向自己（307→自身 = 循环）
- **本地复现要用生产同款条件**，否则复现不出来会误判"代码没问题":
  - 构建期 bug：用 Docker 里同款环境变量本地构建（`NEXT_PUBLIC_API_URL="" npm run build`）
  - 运行时 bug：用同款绑定地址/请求头启动（`HOSTNAME=127.0.0.1 ... -H "Host: <外部域名>"`）
  - 本机默认值常和生产不同（如 dev 模式所有页面动态渲染、HOSTNAME 默认 0.0.0.0），
    正是这种差异掩盖了 bug——**刻意对齐生产的那个变量**。
- **确认你看的是部署版本**：怀疑"代码已经修了但没生效"时，读线上实际产物
  （压缩后的 JS bundle、`prerender-manifest.json`、容器内文件），用 `git log -S "<代码片段>"`
  追这段代码何时引入、HEAD 与线上是否一致。
- 等待长任务（构建/下载）用 `run_in_background` + `Monitor`/`until ... grep` 轮询，不要 `sleep` 链。

产出：一句话可复现步骤 + 实际观测（状态码/响应头/日志），作为后续诊断的锚点。

## 阶段 2 · 定位根因（沿代码追到机制）

不要停在"复现了"。要回答"为什么会这样"。

操作：
- **从症状反向追代码链路**：`Grep` 报错文案、相关函数名、路由定义、中间件、配置默认值，并行读多个文件。
  把"请求进来 → 中间件 → 路由 → 服务 → 写缓存/响应"整条链走一遍，找出在哪一步、什么条件下偏离预期。
- **优先怀疑"边界/默认值/吞异常"三类**，它们最容易在生产暴露而本地不暴露：
  - 被 try/catch 吞掉的、本应传播的信号（框架靠抛异常切换行为时，吞掉=行为退化）
  - 必填参数漏传、新增参数后旧调用点没跟上（→ 运行时 TypeError，常被全局兜底成 500）
  - 第三方/CDN 只接受固定取值（固定宽度、固定枚举），代码却动态拼了非法值（→ 404/400 静默回退）
  - 反代/隧道后真实 IP、Host、协议丢失或可被客户端伪造（→ 限流/鉴权退化或被绕过）
  - 内部 origin / base URL：服务端用了相对地址或绑定了具体 hostname，导致跨域重入或挂死
- **机制对不上就别下结论**：复现现象与你的假设要严丝合缝。若用户假设"X-Real-IP 导致全局限流"，
  但代码实际读的是 XFF 首元素，就要纠正机制，并检查这是否引出**更严重**的问题（如可伪造绕过）。
- 实在核实不了的（依赖线上数据、外部服务真实返回），**明说"代码里核实不到"**，给出验证它的具体命令，不要假装验证过。

产出：机制级根因陈述——"哪一行 + 什么条件 + 为什么 + 为什么本地没暴露"。

## 阶段 3 · 修复（最小改动 + 扫同类拷贝）

操作：
- **改根因点，不打补丁**：在信号源头修（去掉吞异常、补全参数、取整到合法值、还原真实 IP），
  而不是在下游到处加 if 兜。
- **立刻 `grep` 同一错误模式**——这是本流程最容易省掉也最容易翻车的一步：
  - 同一个工具函数被多处复制（`client.ts` 改了，`raw-client.ts` 往往有一份一样的）
  - 同一个新参数有多个调用点（漏一个就还会 500）
  - 同一个 IP/Host 提取逻辑散落在中间件、路由、错误处理多份
  - 改了缓存 key 的构成，**写入、读取、失效（invalidate）、定时同步**四处都要同步改，否则留脏数据
- **保留旧行为做平滑过渡**：改缓存 key 时，失效逻辑同时删新旧两种 key；改默认值时注释写清为什么。
- **加防御而非删功能**：服务端 fetch 一律带超时（`AbortSignal.timeout`），避免任何外部不可达时挂死整条流程。
- 跨平台/环境差异顺手修掉（如脚本里 `cp -r` 在 Windows cmd 必失败 → 改 Node `fs.cpSync`）。

产出：改动文件清单 + 每处为什么改。改前用对应语言的语法/类型检查过一遍（`py_compile`、`tsc`、build）。

---

## 阶段 4 · 验证（在暴露 bug 的同款条件下）

操作：
- **用阶段 1 复现 bug 的同款条件重跑**，确认现象消失——而不是用一个更宽松的本地条件"看起来好了"。
  （构建期 bug 就用同款环境变量重新 build；循环 bug 就用同款 Host 头重新 curl。）
- **正反都验**：坏的 case 现在好了 + 好的 case 没被改坏 + 边界 case（404/空数据/未登录）行为正确。
- **对照矩阵**：多输入维度（locale × 路径 × 登录态）列一张小表逐格验，别只验你改的那一格。
- 确认修复**进入了产物**：build 成功的页数、bundle 里能 grep 到新逻辑、prerender manifest 条目正确。
- 能端到端就端到端（真实端点/headless 截图）；不能就给出用户可自己跑的验证命令。

产出：验证结果（最好是表格），明确"通过/未通过"，不含糊。

---

## 阶段 5 · 交付与收尾

操作：
- **结论先行**：一句话说清根因是什么、改了哪几处、为什么之前没发现（本地与生产的差异）。
- **部署提醒**：指出生效的前提——哪些东西是 `COPY` 进镜像/构建期固化的，**改完必须重新 build + 重启**
  才生效；配置在哪一层（这个仓库里 / 仓库外的反代）。
- **诚实边界**：写清残留限制（"构建容器无后端故首页统计为 0 是设计行为"、"国旗图依赖访问者外网"、
  "这个改动拦得住 X 拦不住 Y"），以及顺带发现但未处理的问题。
- **更新记忆**：把"非显而易见的坑 + 根因 + 排查/验证技巧"写进 auto-memory
  （`MEMORY.md` 一行指针 + `project_*.md` 详情，按"坑 N（日期已修）"追加），便于下次复用。
  只记非显而易见的，代码/git 已记录的不重复记。

## 快速检查清单

```
□ 复现了吗？是从真实环境取证、还是只看了代码就猜？
□ 本地复现用的是生产同款条件（env/header/host/构建产物）吗？
□ 根因是机制级的吗？能说清"哪行+什么条件+为什么+为什么本地不暴露"？
□ 复现现象和我的假设严丝合缝吗？需要推翻表面判断或用户假设吗？
□ grep 过同一 bug 的其他拷贝/其他调用点了吗？
□ 改缓存 key/参数/IP 提取时，相关的读写/失效/同步/兜底都跟着改了吗？
□ 验证用的是暴露 bug 的同款条件？正反+边界+多维矩阵都验了？
□ 修复确认进产物了？给了部署提醒（要重新 build）？
□ 交付说清了根因、为什么之前没发现、残留边界？
□ 非显而易见的坑写进 memory 了？
```
